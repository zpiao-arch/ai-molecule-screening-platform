## 1. 交付的"更新后闭环"到底是什么

按用户诉求"**降低通量但确保跑通、最后速度快、然后冻结**"，本次交付的闭环 = **L2 检索粗筛 + 对接精排(LE 校正) 的级联**，并对唯一有受体的靶点 NA(3TI6) 真正端到端跑通：

- **Stage1 检索粗筛（高吞吐）**：哈希 L2（`Layer2BindingDB`，检索域最优）对 100k 全库一次性向量化打分，取每靶 top-N。
- **Stage2 对接精排（低吞吐但跑通）**：对 top-N 候选并行跑 `smina`(AutoDock Vina 打分) + `obabel`(3D, pH 7.4)，LE = affinity / HAC 校正 Vina 尺寸偏好。
- **Stage3 级联评估**：LE 归一化分数与 L2 分数融合，输出每靶 AUC / EF@1% / recall@1%。

### 本次运行配置
| 项 | 值 |
|---|---|
| 库规模 | 100,000 分子（`lib_feats` 100k×520） |
| 靶点 | CHEMBL2051（NA，受体 3TI6，当前环境**唯一**有受体的靶点） |
| TOPN | 250（对接候选上限） |
| WORKERS × cpu | 5 × 2 = 10 核吃满 |
| exhaustiveness | 4 |
| dock_timeout | 50s（超时即丢弃，不阻塞） |
| HAC 上限 | ≤50（超大多肽自动跳过） |
| 总耗时 | **845.6s ≈ 14 分钟** |

## 2. 核心结果

| 场景 | L2 基线 AUC | 级联(LE) AUC | 增益 |
|---|---|---|---|
| **平衡集**（18 活性 + 100 诱饵，苹果对苹果） | 0.413 | **0.765** | **+0.352** |
| **全库部署**（L2 top-250 → LE 重排） | 0.485 | 0.500 | +0.015（无效） |

辅助指标：
- 平衡集已对接活性：13/18（其余 5 个因 HAC>50 或对接超时未得分）
- 全库部署级联：`n_actives_in_topN = 0`（**18 个活性无一落入 L2 top-250**）
- EF@1% / recall@1%：L2 与级联**均为 0.0**（100k 库 top1%=1000 位，活性既少又未进 top-250）

## 3. 关键结论（诚实口径）

### ✅ 结论①：②对接精排增益**可复现、可冻结**
平衡集 L2 0.413 → 级联 0.765，与早期 NA demo（0.462→0.756）高度一致（甚至略高）。证明 `smina + obabel + LE 校正` 对 NA 是**真实信号**，不是偶然或过拟合。这是本次闭环最重要的正向结论。

### ⚠️ 结论②：部署级联的断点在 **L2 召回**，不是对接模块
全库部署级联 AUC 仅 0.500，根因是 `n_actives_in_topN = 0`——**18 个活性分子全部排在 L2 top-250 之外**。L2（哈希）在 NA 检索域本身弱（0.485≈随机），把活性分子推到了粗筛窗口外，导致下游对接精排"英雄无用武之地"。
→ 这是"级联方法有效，但初筛召回不足"的典型现象：**对接模块没问题，瓶颈在它上游的 L2 粗筛**。

### 📌 结论③：EF/recall 全为 0 是规模函数，非新问题
100k 库 top1% = 1000 位；活性分子既少（18）又没进 top-250，EF/recall 必然塌 0。再次印证：**大通量评测必须以 AUC 为主指标**，EF/recall 不可跨规模比较。

## 5. 限制与扩展路径

- **受体约束（硬限制）**：当前环境仅 NA(3TI6) 有受体 pdbqt + docking box，故对接精排只作用在 NA。多靶点扩展需补充各靶点的受体结构与盒子坐标。
- **部署断点修复方向**（任选）：
  1. 提高 `TOPN`（如 1000–2000）让活性进入对接候选窗口——直接缓解 `n_actives_in_topN=0`；
  2. 用更强的初筛（如多靶点协同投票 / 分子相似性召回）替代纯 L2 排序；
  3. 对 L2 召回失败靶点，改用 DUD-E 式属性匹配诱饵做对接复测，避免被背景库淹没。
- **性能**：14 分钟跑完 100k 库检索 + NA 对接，满足"可用前提下速度快"的要求。
